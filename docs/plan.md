# 通知系统开发计划

## 1. 目标

当前通知系统的后端基础能力已经具备，但前端闭环、通知展示质量、部署一致性和联调流程仍不完整。本计划的目标是：

1. 先把现有“通知”能力做成完整可用的产品闭环。
2. 在不打乱现有业务链路的前提下，按前后端分工并行推进。
3. 明确各阶段依赖关系，避免前后端各做各的，最后难以合拢。

本计划默认本轮只做“通知中心”，不把“私信系统”混入本轮范围。

## 2. 总体执行顺序

### Phase 1：先完成现有通知能力闭环

目标：不大改协议，优先把已有通知数据和接口真正用起来。

执行原则：

1. 后端接口尽量复用现有能力，不先扩协议。
2. 前端优先补齐“未读、已读、分类展示、入口、交互”。
3. 联调时先确认列表、未读数、实时推送、已读状态一致。

### Phase 2：提升通知内容质量

目标：让通知从“系统提示”升级为“可理解、可点击、可跳转”的通知卡片。

执行原则：

1. 后端补 VO，不直接把 DO 暴露给前端。
2. 前端在 UI 上承接 richer payload。
3. 这一阶段允许前后端并行，但以前端消费协议为界面收口点。

### Phase 3：工程加固与部署一致性

目标：解决配置硬编码、WebSocket 稳定性、联调与回归问题。

执行原则：

1. 统一 HTTP / WS 地址配置。
2. 补联调检查项和自动化验证。
3. 把最终可上线的运行方式和回归流程固化。

## 3. 前后端分工总览

### 前端负责

1. 通知中心页面闭环。
2. Header 入口和未读态展示。
3. WebSocket 消费、列表实时更新、已读交互。
4. 后续 richer payload 的展示、跳转和 UI 收口。

### 后端负责

1. 保持现有通知触发链路稳定。
2. 提供可消费的通知协议升级版本。
3. 保证未读数、已读状态、实时推送语义一致。
4. 统一部署配置和联调脚本。

## 4. 打勾完成说明

后续执行本计划时，建议将每一条可执行任务改写为 Markdown checklist，并按以下规则维护：

1. 未开始：`[ ]`
2. 已完成并验证：`[x]`
3. 不再执行或延期：保留原任务，并在后面标注“延期”或“移出本轮”原因，不直接删除。

每一项被打勾后，建议补一行“完成说明”，至少写清以下信息：

1. 完成时间
2. 完成人或负责端
3. 改动范围
4. 验证方式
5. 是否存在遗留问题

建议格式如下：

```md
- [x] Header 显示总未读 badge
      完成说明：2026-03-27，前端完成；改动 `Header.vue` 和通知状态管理；已通过手工联调验证；无阻塞遗留问题。
```

如果一个阶段整体完成，也建议在该阶段标题下增加阶段性完成说明，记录：

1. 本阶段是否达成目标
2. 实际完成范围
3. 未完成项
4. 是否允许进入下一阶段

## 5. 按阶段拆分的开发计划

## Phase 1：通知中心闭环

### 优先级

- P0：必须先完成
- P1：建议本阶段一起完成
- P2：可放到下一阶段

### 前端任务

#### P0

- [ ] 将“我的消息”页面明确收口为“通知中心”。
- [ ] 新增“新粉丝”通知 tab，统一承接 `followNotifications`。
- [ ] 接入未读数展示。
      完成标准：Header 显示总未读 badge；通知 tab 显示分类未读态或总未读态。
- [ ] 实现通知已读流转。
      完成标准：点击单条通知时调用 `markRead`；提供“全部标记为已读”；已读/未读样式有明确区分。
- [ ] 处理实时通知插入后的未读状态。
      完成标准：WebSocket 新通知进入列表顶部；未读数同步增加。
- [ ] 去掉“私信已完成”的假象。
      完成标准：当前没有私信后端时，标记为“暂未开放”或从本轮通知中心中移出。

#### P1

- [ ] 通知列表支持空状态、加载状态、错误状态。
- [ ] 点击通知后根据 `targetType/targetId` 做基础跳转。
      完成标准：帖子通知跳帖子详情；用户通知跳用户主页或粉丝页。
- [ ] 前端对实时通知和首次加载通知去重，避免重复插入。

#### P2

- [ ] 前端分类未读数单独展示。
- [ ] 通知筛选和分页优化。

### 后端任务

#### P0

- [ ] 保持现有通知接口稳定。
      完成标准：`GET /notifications`、`GET /notifications/unread-count`、`POST /notifications/{id}/read`、`POST /notifications/read-all` 可正常使用。
- [ ] 确认点赞、回复、关注三类通知语义稳定，且不会给自己发通知。
- [ ] 确认实时推送与落库顺序一致。
      完成标准：先落库，再推送。
- [ ] 确认 `isRead` 与未读数统计一致。

#### P1

- [ ] 为前端跳转补充最低限度规则说明。
      完成标准：`targetType=POST` 跳帖子详情；`targetType=REPLY` 跳帖子详情并定位回复；`targetType=USER` 跳用户相关页。
- [ ] 增加通知接口返回字段说明文档，明确当前前端可依赖的字段。

#### P2

- [ ] 如果现有 `GET /notifications` 无法满足前端首屏需要，再考虑补筛选参数。

### Phase 1 合拢点

前后端在这一阶段的合拢标准：

- [ ] 用户能在 Header 看见未读通知数。
- [ ] 用户能在通知中心看见“赞 / 回复 / 新粉丝”三类通知。
- [ ] 新通知产生后链路完整。
      完成标准：数据落库；未读数增加；WebSocket 收到实时通知；前端列表顶部新增。
- [ ] 点击单条已读、全部已读后状态一致。
      完成标准：列表状态更新；未读数同步归零或减少；刷新页面后状态一致。

### Phase 1 完成后的验收成果

完成 Phase 1 后，应能明确交付以下成果：

- [ ] 一版可用的“通知中心”页面，而不是半成品“消息中心”。
- [ ] Header 未读 badge 可用，且与列表状态一致。
- [ ] 点赞、回复、新粉丝三类通知均可查看。
- [ ] 单条已读、全部已读行为可操作且可回刷验证。
- [ ] 一份 Phase 1 联调记录已产出。
      完成标准：包含列表拉取验证、未读数验证、WebSocket 实时通知验证、已读状态验证。
- [ ] 一份 Phase 1 完成说明已产出，明确是否允许进入 Phase 2。

## Phase 2：通知协议升级与展示增强

### 目标

把通知从“固定文案”升级为“带上下文信息的用户通知卡片”。

### 前端任务

#### P0

- [ ] 适配新的 `NotificationVO` 展示结构。
- [ ] 在通知卡片中展示完整上下文信息。
      完成标准：展示操作人昵称、操作人头像、目标标题或摘要、更明确的动作文案。
- [ ] 通知卡片支持整体点击跳转。

#### P1

- [ ] 支持更清晰的通知样式差异。
      完成标准：赞、回复、关注三类样式可区分。
- [ ] 对回复类通知增加上下文摘要。
- [ ] 对帖子类通知展示帖子标题。

### 后端任务

#### P0

- [ ] 新增 `NotificationVO`，不再直接返回 `NotificationDO`。
- [ ] 补齐通知展示核心字段。
      完成标准：至少包含 `actorName`、`actorAvatar`、`targetTitle`、`targetPreview`、`targetUrl` 或前端可推导的跳转信息。
- [ ] 列表接口与 WebSocket 推送保持字段结构一致，避免前端维护两套模型。

#### P1

- [ ] 在服务层统一组装通知展示数据，避免控制器拼接。
- [ ] 对缺失目标数据的历史通知定义降级策略。
      完成标准：明确目标不存在时是否仍展示、是否隐藏跳转。

### Phase 2 合拢点

- [ ] 列表接口和 WS 推送结构统一。
- [ ] 前端不再依赖固定 `content` 文案作为唯一展示来源。
- [ ] 通知具备“谁、对什么、做了什么、点进去去哪”的完整信息。

### Phase 2 完成后的验收成果

完成 Phase 2 后，应能明确交付以下成果：

- [ ] 后端输出统一的 `NotificationVO` 协议。
- [ ] 列表接口和 WebSocket 推送字段结构一致。
- [ ] 前端通知卡片可展示完整信息并支持跳转。
      完成标准：包含操作人信息、目标信息、动作信息、可点击跳转。
- [ ] 一份前后端字段对齐说明已产出，明确最终通知协议。
- [ ] 一份历史通知降级策略说明已产出，明确异常或缺失数据时前端如何展示。
- [ ] 一份 Phase 2 完成说明已产出，明确是否允许进入 Phase 3。

#### P1

### 后端任务

#### P0

- [ ] HTTP 和 WS 不再依赖本地硬编码端口。

      完成标准：覆盖 smoke check、ws + redis notification check。
- [ ] 一份最终联调报告已产出，说明通知系统已达到可持续回归状态。
- [ ] 一份 Phase 3 完成说明已产出，明确本轮通知系统是否收口完成。

## 6. 推荐并行开发方式

### 第一轮并行

前端先做：

1. 通知中心 UI 重构。
2. Header 未读 badge。
3. 已读交互和本地状态管理。

后端同时做：

1. 梳理现有通知字段契约。
2. 确认已读、未读、实时推送语义稳定。
3. 补接口说明和联调样例。

这一轮的目标不是升级协议，而是把现有系统跑通。

### 第二轮并行

前端先定义通知卡片需要的展示字段。

后端据此输出 `NotificationVO` 和统一推送结构。

然后前端完成新协议适配和点击跳转。

### 第三轮并行

前端处理地址配置、重连、状态恢复。

后端处理部署配置、联调脚本、文档。

最终以联调回归为收口。

## 7. 最终合拢与验收标准

项目最终以以下标准验收：

1. 点赞、回复、关注都能产生通知。
2. 通知能落库、可分页拉取、可统计未读数。
3. WebSocket 能实时收到通知。
4. Header 未读数、通知列表、已读状态三者一致。
5. 通知列表包含“赞 / 回复 / 新粉丝”三类可见入口。
6. 通知卡片具备可读上下文和可点击跳转能力。
7. 开发环境和部署环境均无 HTTP / WS 硬编码依赖。
8. 现有 smoke 与 ws+redis 检查脚本可以作为回归基线。

## 8. 明确不在本轮范围内

以下内容不建议混入本轮，以免分散通知系统收口：

1. 完整私信系统后端建设。
2. 邮件、短信、站外推送。
3. 复杂通知偏好策略中心。
4. 大规模消息队列化改造。

这些内容可以在通知中心闭环完成后另开阶段规划。
